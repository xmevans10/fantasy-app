## 1. Era-multiplier statistical analysis (make `.eraPoints` real)

The scaffolding exists but has never been validated against data:

- `BallIQ/Data/stat_baselines.json` — 2,411 rows of `(sport, position, stat, year) → mean/std/count`,
  generated by `tools/ingest/baselines.py` (5+ sample minimum, position-pooled per year).
- `ScoringRule.eraIndex()` (`BallIQ/Models/ScoringRule.swift`) — volume index `globalMean / eraMean`,
  applied per stat by the `.eraPoints` normalization. `StatBaselines.globalMean` is the
  count-weighted cross-era mean. **The era-adjust lever is currently OFF the creation UI surface.**

**The task:** run a real statistical analysis over the full catalog (the `player_seasons` pull the
pipeline already makes — see `tools/ingest/main.py` / providers) to answer:

1. Is per-stat `globalMean / eraMean` the right multiplier shape, or does a single per-year
   *fantasy-total* index (era PPR mean vs global PPR mean, per position) behave better? Volume-based
   diffs are the target: passing volume exploded post-2004 rules changes; receptions ballooned in
   the 2010s; NBA pace cratered in the 90s and recovered.
2. What do the multipliers actually look like by year? Sanity-check famous seasons: does 1984 Marino
   or 1995 Rice era-adjust ABOVE a comparable modern line? Document a table of indices in the PR.
3. Where is the data too thin to trust (the `count >= 8` fallback in `ScoringRule`)? Tighten or
   loosen with evidence.

Then wire the winner in: pipeline-side (a `*_era` scale variant or a flag on `Theme`, so a daily
theme like "best seasons of all time, era-adjusted" is possible) and creator-side (surface the
existing `eraAdjusted` toggle in `CreateKeep4View`'s scoring card with a one-line explanation).
Keep raw points as the default; era-adjust is an explicit mode. Keep the Swift/Python
implementations in exact parity, with locked-value tests on both sides.

**Success:** analysis artifact (script under `tools/ingest/` + findings in the PR/commit message);
`.eraPoints` produces defensible orderings on the sanity set; toggle visible in creation; parity
tests green.

## 2. Community templates mirror daily themes — same exact data structure

Today the daily pipeline and the creation flow build puzzles through different structures:
`themes.py` `Theme` (positions, min_stats, filters, columns, pool_cap) vs `CreateKeep4View`'s
free-form `ScoringRule` + facet search. The user wants one shared shape.

**The task:** define the template structure ONCE and use it on both sides. Concretely:

- Extract the shared shape (scale key, position set, stat columns w/ formats, inclusion floors)
  into data both sides consume. Options: ship a JSON export of `KEEP4_THEMES` into the app bundle
  (pipeline already bakes `keep4_puzzles.json`; a `keep4_themes.json` sibling is natural), or a
  Swift mirror locked by a parity test like `GradeFormulaTests`. Prefer the JSON export — one
  source of truth, no drift.
- `CreateKeep4View` presents those themes as creation starting points: picking one sets the scoring
  rule, position filters, and card stat columns to exactly what the daily pipeline uses; the author
  then curates the 8 picks. Free-form facet search stays for discovery, but the published card's
  `stats` columns come from the theme (today they're derived from the first 3 scoring terms —
  see `cards()` — which is the drift the user is pointing at).
- Published community `content` must be indistinguishable in shape from a daily row (it nearly is —
  `Keep4Puzzle` Codable — the gap is column selection + grade provenance).

**Success:** a community puzzle created from a theme renders cards identical in structure to that
theme's daily puzzles; one definition of the template data; parity/round-trip test proving the
Swift-decoded theme matches `themes.py`.
